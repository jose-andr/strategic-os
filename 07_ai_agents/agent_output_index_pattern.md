# Agent Output Index Pattern

## Purpose

This file defines a lightweight index pattern for saved Strategic OS agent outputs.

The index helps make agent-generated material easier to find, review, reuse, retire or convert into more durable Strategic OS knowledge.

It is not intended to become a heavy register or administrative burden.

## Core Principle

Only index outputs that are worth finding again.

The goal is to preserve useful thinking, not catalogue every interaction with an agent.

An output should be indexed when it may support:

* a future decision
* strategic opportunity review
* stakeholder pattern recognition
* career evidence
* reusable learning
* project continuity
* analytical caveat tracking
* quarterly Strategic OS review

## When To Use An Output Index

Use an output index when a folder contains multiple saved agent outputs that may need to be reviewed over time.

Good candidates:

* `01_career/`
* `02_strategic_opportunities/`
* `03_decision_briefs/`
* `05_lessons_learned/`
* `06_stakeholder_patterns/`
* `08_projects/`
* `09_thought_leadership/`
* `11_analytics/[domain]/`

An index is especially useful when outputs relate to:

* decisions
* opportunities
* stakeholder alignment
* career evidence
* lessons learned
* reusable frameworks
* recurring projects
* quarterly reviews
* governed analytics
* metric caveats
* source rules
* Databricks / Genie review prompts

## When Not To Use An Output Index

Do not create or update an index for:

* temporary notes
* low-value drafts
* one-off outputs with no future value
* routine daily briefings that are not reused
* admin task lists with no durable value
* follow-up trackers that are only temporary
* sensitive material that should not be stored
* folders with only one or two stable files
* outputs already captured cleanly in a project file or index
* material that would create more maintenance burden than retrieval value

## Recommended Index File Name

Use:

`agent_output_index.md`

Place it inside the folder it indexes.

Examples:

* `03_decision_briefs/agent_output_index.md`
* `06_stakeholder_patterns/agent_output_index.md`
* `08_projects/project-name/agent_output_index.md`
* `11_analytics/service-account/agent_output_index.md`

## Standard Index Structure

Use this structure:

# Agent Output Index

## Purpose

This index tracks saved agent outputs in this folder that may be useful for review, reuse or future decision-making.

## Index

| Date       | Agent      | Output Type | Topic | File         | Status | Notes      |
| ---------- | ---------- | ----------- | ----- | ------------ | ------ | ---------- |
| YYYY-MM-DD | Agent Name | Output Type | Topic | file-name.md | Active | Short note |

## Status Values

Use simple status values:

* Active
* Draft
* Superseded
* Archived
* Reusable
* Review

## Review Notes

Use this section for lightweight observations about patterns emerging across saved outputs.

## Maintenance

Review this index during the monthly or quarterly Strategic OS cadence.

## Field Definitions

### Date

The date the output was created or saved.

Use ISO format:

`YYYY-MM-DD`

### Agent

The agent that produced or supported the output.

Examples:

* Sensemaking Agent
* Stakeholder Journey Agent
* Shipping Coach
* Career Architect
* Chief of Staff Agent
* Domain Analytical Agent
* Service Account Analytics Agent
* Databricks / Genie Domain Agent

### Output Type

The kind of output saved.

Examples:

* Daily Briefing
* Weekly Operating Review
* Admin Task List
* Follow-Up Tracker
* Decision Brief
* Stakeholder Alignment Assessment
* Shipping Recommendation
* Opportunity Review
* Career Guidance
* Promotion Readiness Assessment
* Sensemaking Summary
* Lesson Learned
* Analytical Summary
* Metric Caveat Note
* Source Rules
* Agent Review Prompt
* Genie Context
* Data Quality Note
* Consistency Check

### Topic

A short description of the subject.

Examples:

* channel strategy
* stakeholder alignment
* operating cadence
* service account review
* business design positioning
* decision quality
* customer demand analysis
* support CSAT caveats
* executive metrics review
* data quality pattern

### File

The file name of the saved output.

Use the naming conventions in:

`07_ai_agents/agent_file_naming.md`

### Status

The current usefulness of the output.

Use:

* Active: currently relevant
* Draft: unfinished or provisional
* Superseded: replaced by a later output
* Archived: retained for history only
* Reusable: useful as a pattern or reference
* Review: needs human review before reuse

### Notes

Use brief notes only.

Good notes:

* Supports quarterly review
* Useful decision logic
* Reusable stakeholder framing
* Superseded by later brief
* Candidate for template
* Data caveat to check before reuse
* Good example of source-rule handling

Avoid long explanations.

## Chief of Staff Output Indexing

Most Chief of Staff outputs should remain temporary working artefacts.

Do not index routine daily coordination material unless it creates durable value.

Index Chief of Staff outputs only when they create:

* decision context
* reusable operating learning
* project continuity
* stakeholder follow-up evidence
* strategic opportunity signals
* career evidence
* unresolved risks that need review
* patterns worth checking during weekly or quarterly cadence

Examples worth indexing:

* a weekly operating review that identifies a recurring blocker
* a follow-up tracker that records a durable dependency pattern
* a daily briefing that led to a decision brief
* an admin task list that revealed a hidden decision or governance risk

Examples not worth indexing:

* a routine daily briefing
* a completed admin task list
* temporary reminders
* low-value scheduling notes
* one-off follow-up lists with no reuse value

## Domain Analytical Agent Indexing

Domain analytical outputs require stronger review discipline because they may influence decisions, reporting or interpretation of governed data.

Index analytical outputs when they:

* define source rules
* define metric caveats
* support a decision brief
* document a recurring data quality issue
* clarify governed versus contextual metrics
* create reusable Genie prompt context
* identify an analytical assumption that must be checked again
* provide decision-support evidence that may be reused

Do not index:

* raw query outputs
* copied governed datasets
* exploratory analysis with no reuse value
* sensitive or customer-level records
* temporary Databricks or Genie interaction notes
* unsupported interpretations

## Review Cadence

Review indexes during:

* weekly inbox triage where relevant
* monthly opportunity review
* monthly lessons learned review
* quarterly repository hygiene review
* quarterly agent performance review
* analytics domain review where governed data or caveats are involved

Do not review every index every week.

Use judgement.

## Privacy Guidance

Indexes should not expose sensitive context.

Do not include:

* customer names
* confidential stakeholder names
* raw organisational source references
* sensitive internal issues
* private employee matters
* governed dataset details
* confidential system extracts

Use abstract, reusable labels.

Instead of:

`Jane Smith conflict`

Use:

`senior sponsor alignment risk`

Instead of:

`customer complaint case 12345`

Use:

`complaint handling escalation pattern`

Instead of:

`raw Databricks extract for service account issue`

Use:

`service account metric caveat`

## Success Measures

The index pattern is working when:

* useful outputs are easy to find
* related outputs can be compared
* reusable thinking is visible
* old drafts do not clutter the system
* quarterly reviews are easier
* career evidence is easier to assemble
* decision history is easier to trace
* analytical caveats are easier to reuse
* sensitive details are not exposed
* temporary Chief of Staff outputs are not over-indexed

## Failure Modes

The index pattern is failing if:

* every minor output is indexed
* indexes become more work than value
* sensitive details appear in index rows
* status values are ignored
* outdated outputs are treated as current
* the index is not reviewed
* file names and index entries do not match
* routine daily briefings are preserved without reuse value
* raw analytical outputs are indexed instead of reviewed summaries
* Databricks / Genie caveats are separated from the outputs they affect

## Maintenance

During quarterly repository hygiene review:

* remove low-value entries
* mark superseded outputs
* archive outdated material
* update missing links or file names
* identify reusable patterns
* promote valuable outputs into frameworks or templates
* check that analytical caveats are still valid
* check that Chief of Staff outputs are not creating unnecessary clutter

## Minimal Use Rule

When in doubt, keep the index light.

A useful index is better than a complete index.

The index should help Strategic OS remain usable, not turn it into administration.
