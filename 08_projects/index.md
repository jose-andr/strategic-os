# Projects Index

## Purpose

This index tracks project folders stored in `08_projects/`.

Use it to make project work easier to find, review, connect to decisions, convert into lessons learned and translate into career evidence.

The index supports the Strategic OS North Star:

> Increase strategic opportunities by helping leaders make better strategic decisions.

This is not a delivery tracker.

It is a lightweight navigation aid for project knowledge stored in Strategic OS.

## What To Index

Index projects that:

- have a dedicated project folder
- include multiple outputs or decisions
- create reusable learning
- support strategic opportunities
- involve stakeholder alignment
- may generate career evidence
- may create templates or frameworks
- may need future review or reference

Do not index:

- single tasks
- minor notes
- temporary ideas
- sensitive work that should not be stored
- projects that only belong in official organisational systems
- delivery tickets that should remain in Jira or another delivery tool

## Status Values

Use simple status values.

| Status | Meaning |
| --- | --- |
| Captured | Possible project, not yet shaped |
| Shaping | Being clarified or scoped |
| Active | Current project work |
| Paused | Temporarily stopped |
| Completed | Project work is complete |
| Archived | Retained for reference only |
| Converted | Converted into framework, template, lesson, career evidence or thought leadership |

## Field Guidance

## Project

Use a clear, privacy-safe project name.

Good examples:

- channel-strategy
- service-account-analytics
- operating-cadence
- stakeholder-alignment-patterns
- strategic-os-qa

Avoid:

- confidential project names
- customer identifiers
- named stakeholder issues
- sensitive internal issue names
- vague names like `misc-project`

## Folder

Use the exact folder path.

Pattern:

`08_projects/project-name/`

Example:

`08_projects/channel-strategy/`

## Status

Use one of the standard status values.

## Strategic Value

Summarise why the project matters strategically.

Consider whether the project helps:

- improve decision quality
- clarify complexity
- align stakeholders
- improve customer outcomes
- reduce operational friction
- build reusable capability
- create career evidence
- support strategic opportunities

## Key Output

Name the most useful output or artefact.

Examples:

- decision brief
- system map
- stakeholder alignment assessment
- service blueprint
- analytics interpretation note
- project lesson
- framework
- template
- career evidence note

## Next Review

Use the next date the project should be reviewed.

Use ISO format:

`YYYY-MM-DD`

Leave blank if no review is needed.

## Recommended Project Folder Pattern

Most project folders should follow:

`08_projects/project-name/README.md`

`08_projects/project-name/context.md`

`08_projects/project-name/decisions.md`

`08_projects/project-name/stakeholder-notes.md`

`08_projects/project-name/outputs/`

`08_projects/project-name/lessons-learned.md`

`08_projects/project-name/archive/`

Use:

`04_frameworks/project_folder_pattern.md`

for the full structure.

## Review Rhythm

Review this index during:

- weekly project movement review, where relevant
- monthly strategic opportunities review
- monthly lessons learned review
- quarterly Strategic OS QA assessment
- project closure review

Do not review it more often than it creates value.

## Project Conversion Paths

Project work may later become:

- a decision brief in `03_decision_briefs/`
- a lesson learned in `05_lessons_learned/`
- a stakeholder pattern in `06_stakeholder_patterns/`
- a reusable framework in `04_frameworks/`
- a reusable template in `10_templates/`
- career evidence in `01_career/`
- a thought leadership seed in `09_thought_leadership/`
- an analytics pattern in `11_analytics/`
- a strategic opportunity in `02_strategic_opportunities/`

## Privacy Guidance

Project indexes should not expose sensitive project detail.

Do not include:

- customer-level records
- confidential stakeholder names
- private stakeholder commentary
- governed dataset details
- sensitive internal issues
- private employee matters
- confidential source extracts
- raw operational records

Use safe abstractions.

Instead of:

`Jane Smith complaints escalation issue`

Use:

`complaint-handling-escalation-pattern`

Instead of:

`customer case 12345 service failure`

Use:

`service-continuity-risk`

Instead of:

`Team X data quality failure`

Use:

`analytics-source-confidence-review`

## Quality Check Before Indexing

Before adding a project to the index, check:

- Does this project have or need a dedicated folder?
- Is the project name privacy-safe?
- Is the status accurate?
- Is the strategic value clear?
- Is there a useful output or expected output?
- Is the next review date useful?
- Does the project belong in Strategic OS rather than only in official systems?
- Could this project create reusable learning, decision support or career evidence?

## Success Measures

This index is working when:

- active projects are easy to find
- project status is visible
- important outputs are easier to locate
- projects connect to decisions, lessons and opportunities
- project closure is easier
- career evidence is easier to identify
- project folders do not become isolated knowledge silos

## Failure Modes

This index is failing if:

- every small task becomes a project
- project statuses are not updated
- sensitive project detail appears in the index
- projects have no clear strategic value
- project folders exist but are not reviewed
- completed projects are never closed or archived
- project outputs are not converted into reusable knowledge
- the index becomes harder to maintain than useful

## Minimal Use Rule

Index only projects that need to be found, reviewed or reused.

A useful project index is better than a complete project register.
